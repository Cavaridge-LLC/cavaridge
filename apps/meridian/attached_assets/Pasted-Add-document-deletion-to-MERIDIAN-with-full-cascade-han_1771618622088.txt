Add document deletion to MERIDIAN with full cascade handling. When a document 
is removed, EVERYTHING downstream that was generated from that document must 
also be cleaned up, and all scores must be recalculated.

This is one of the most important features for data integrity. Build it 
carefully.

How to determine auto-generated findings:
- Check findings where source_documents array contains this document's filename
- OR check findings where description contains "[Auto-detected from image analysis]"
- OR check findings where metadata_json contains this document's ID
- Ideally, add a column "source_document_id" (UUID, nullable, FK to documents) 
  to the findings table to make this relationship explicit. If the column 
  doesn't exist, add it now and backfill it for any findings that were 
  auto-generated from document/image analysis.

How to determine chat citations:
- Search chat_messages where content contains the document's filename
- Return the count (not the messages themselves)

For archive parents (ZIP files):
- Count ALL child documents (documents where parent_archive_id = this doc)
- Include their chunks, findings, etc. in the cascade totals
- The impact is the ZIP + everything inside it

PART 2 — DELETE API WITH CASCADE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create API route DELETE /api/documents/[id] that:

1. Verifies the user has permission to delete documents for this deal 
   (org_owner, org_admin, or analyst with deal access)

2. Runs the entire deletion inside a SINGLE DATABASE TRANSACTION.
   If any step fails, roll back everything.

3. Accepts an optional request body:
   {
     "delete_auto_findings": true,    // default true — remove findings 
                                       // that were auto-generated from this doc
     "delete_children": true,          // default true — if ZIP, delete all 
                                       // extracted files too
     "confirmation": "DELETE"          // required for archive parents with 
                                       // children
   }

DELETION SEQUENCE (order matters):

Step A — Identify all document IDs to delete:
  - The target document itself
  - If it's an archive parent AND delete_children is true: 
    all documents where parent_archive_id = target document ID
  - Collect all these IDs into an array: docs_to_delete[]

Step B — Delete document chunks for ALL docs_to_delete:
  DELETE FROM document_chunks WHERE document_id = ANY(docs_to_delete);
  Record: chunks_deleted count

Step C — Delete or update auto-generated findings:
  IF delete_auto_findings is true:
    Find findings linked to these documents:
    - WHERE source_document_id = ANY(docs_to_delete)
    - OR WHERE source_documents array overlaps with filenames of docs_to_delete
    
    For each finding to delete:
      - Record its pillar_id (we need this for recalculation)
      - Record its severity and deal_id
      - DELETE the finding
    Record: findings_deleted count, affected_pillar_ids[]

  IF delete_auto_findings is false:
    Update the findings to remove the document reference but keep the finding:
    - Set source_document_id = NULL
    - Remove the filename from source_documents array
    - Append to description: "\n[Source document deleted — finding retained for manual review]"
    Record: findings_orphaned count

Step D — Clean up chat citation references:
  Do NOT delete chat messages. Instead, for any chat_message whose content 
  references a deleted document filename, add a metadata flag:
  UPDATE chat_messages 
  SET metadata_json = jsonb_set(
    COALESCE(metadata_json, '{}'), 
    '{stale_citations}', 
    'true'
  )
  WHERE deal_id = (deal of deleted doc)
    AND content LIKE '%' || filename || '%';
  
  If metadata_json column doesn't exist on chat_messages, skip this step.
  Record: chat_messages_flagged count

Step E — Delete document records:
  DELETE FROM documents WHERE id = ANY(docs_to_delete);
  Record: documents_deleted count

Step F — Delete files from storage:
  For each deleted document:
  - Find the file on disk (check the file storage path used by the upload system)
  - Delete the file
  - If the file doesn't exist on disk, log a warning but don't fail
  Record: files_deleted count, bytes_freed

Step G — Update deal counters:
  UPDATE deals SET 
    documents_uploaded = (SELECT count(*) FROM documents WHERE deal_id = $dealId),
    documents_analyzed = (SELECT count(*) FROM documents WHERE deal_id = $dealId 
                          AND extraction_status = 'extracted')
  WHERE id = $dealId;

Step H — Recalculate pillar scores (CRITICAL):
  For each pillar_id in affected_pillar_ids:
  
  a. Count remaining findings by severity for this pillar:
     SELECT severity, count(*) 
     FROM findings 
     WHERE pillar_id = $pillarId AND status != 'remediated'
     GROUP BY severity;
  
  b. Recalculate pillar score using the same formula as deal creation:
     Start at 5.0, then subtract:
     - critical findings: -0.8 each
     - high findings: -0.5 each
     - medium findings: -0.25 each
     - low findings: -0.1 each
     Floor at 1.0 (score cannot go below 1.0)
  
  c. Update the pillar record:
     UPDATE pillars SET 
       score = $newScore, 
       finding_count = (SELECT count(*) FROM findings WHERE pillar_id = $pillarId),
       updated_at = NOW()
     WHERE id = $pillarId;

Step I — Recalculate composite score (CRITICAL):
  After all pillar scores are updated:
  
  a. Get all pillar scores and weights for this deal:
     SELECT score, weight FROM pillars WHERE deal_id = $dealId;
  
  b. Calculate weighted average:
     composite = SUM(score * weight) / SUM(weight)
  
  c. Apply critical penalty factor:
     critical_count = count of findings with severity = 'critical' 
                      AND status != 'remediated' for this deal
     penalty = MIN(critical_count * 5, 30) / 100
     final_score = composite * (1 - penalty)
     Scale to 0-100: final_score = (final_score / 5) * 100
  
  d. Update the deal:
     UPDATE deals SET 
       composite_score = ROUND(final_score, 1),
       updated_at = NOW()
     WHERE id = $dealId;

Step J — Audit log:
  INSERT INTO audit_log (
    id, organization_id, user_id, action, resource_type, resource_id, 
    details_json, created_at
  ) VALUES (
    gen_random_uuid(),
    $orgId,
    $userId,
    'document_deleted',
    'document',
    $documentId,
    jsonb_build_object(
      'filename', $filename,
      'classification', $classification,
      'chunks_deleted', $chunksDeleted,
      'findings_deleted', $findingsDeleted,
      'findings_orphaned', $findingsOrphaned,
      'children_deleted', $childrenDeleted,
      'files_deleted', $filesDeleted,
      'bytes_freed', $bytesFreed,
      'chat_messages_flagged', $chatMessagesFlagged,
      'pillars_recalculated', $affectedPillarNames,
      'score_before', $oldCompositeScore,
      'score_after', $newCompositeScore,
      'was_archive_parent', $isArchiveParent
    ),
    NOW()
  );

RETURN the response:
{
  "success": true,
  "deleted": {

    "files": 12,

    "composite_score_before": 72,
    "composite_score_after": 78.5
  },
  "warnings": [
    "3 chat messages contain references to deleted documents"
  ]
}

PART 3 — BATCH DELETE API
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create API route DELETE /api/documents/batch that accepts:
{
  "document_ids": ["uuid1", "uuid2", "uuid3"],
  "delete_auto_findings": true,
  "delete_children": true,
  "confirmation": "DELETE"
}

This runs the same cascade logic as single delete but for multiple documents 
in one transaction. Aggregate the impact across all documents.

Useful for: "select all unanalyzed images and delete them" or "delete 
everything from this failed ZIP upload."

The "Keep auto-generated findings" checkbox:
- Unchecked (default): findings are DELETED and scores recalculate
- Checked: findings are KEPT but unlinked from the document. The user 
  might want to keep the finding even if the source evidence is gone 
  (e.g., they saw something in a screenshot and added context manually).

The "I understand this cannot be undone" checkbox MUST be checked before 
the Delete button enables.

PART 7 — PERMISSIONS & GUARDRAILS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Who can delete documents:
- platform_owner: any document in any org
- platform_admin: any document in any org
- org_owner: any document in their org
- org_admin: any document in their org
- analyst: documents they uploaded (check uploaded_by = user.id), 
  or any document on deals they have "lead" access to
- integration_pm: documents they uploaded on assigned deals only
- viewer: CANNOT delete anything

Additional guardrails:
- Cannot delete documents on deals in "Closed" stage unless you're 
  org_owner or above. Show: "This deal is closed. Only administrators 
  can modify closed deal documents."
- All deletions are logged to audit_log (see Step J above)
- There is NO undo. The modal makes this clear.
