Create this file. It must:
- Connect to the same database the app uses (import the db config)
- Run ALL deletions inside a single transaction
- Roll back everything if ANY step fails
- Support three CLI modes via flags

USAGE:
  npm run sterilize              → prints warning, does nothing
  npm run sterilize -- --dry-run → shows what WOULD be deleted (counts only)
  npm run sterilize -- --confirm → actually deletes everything

  "sterilize": "node scripts/sterilize-production.js"

Handle all of these:

1. TABLE DOESN'T EXIST YET:
   Before each DELETE, check information_schema.tables. If the table hasn't 
   been created yet (because that prompt hasn't run), skip it and log 
   "Skipping [table] — does not exist". Do NOT crash.

2. COLUMN DOESN'T EXIST:
   If is_platform_user column doesn't exist on users table, fall back to:
   DELETE FROM users WHERE organization_id != (cavaridge_org_id)
   Use a try/catch to detect column errors.

3. CAVARIDGE ORG MISSING:
   If the Cavaridge org doesn't exist, ABORT. Do not proceed. Print:
   "ABORT: Cavaridge, LLC organization not found in database. 
    The platform anchor must exist before sterilization."

4. NO PLATFORM USERS:
   If no users have is_platform_user = true (or no users belong to Cavaridge), 
   ABORT. Print:
   "ABORT: No platform users found. Sterilization would lock out all users."

5. ALREADY CLEAN:
   If dry-run shows 0 rows to delete across all tables, print:
   "Platform is already clean. Nothing to sterilize."
   Exit gracefully.

6. FOREIGN KEY VIOLATIONS:
   If a DELETE fails due to a foreign key not accounted for in the sequence, 
   the transaction rolls back. Log the specific error and which table/FK 
   caused it so it can be debugged.

7. FILE SYSTEM ERRORS:
   If file deletion fails (permissions, locked file), log the error but don't 
   abort. The database cleanup is the critical part. Report any file errors 
   at the end: "Warning: X files could not be deleted"

8. CONCURRENT ACCESS:
   The sterilization does NOT put the app in maintenance mode automatically.
   If you want to prevent users from accessing during sterilization, turn on 
   maintenance mode in Platform Settings FIRST, then sterilize, then turn 
   maintenance mode off.
