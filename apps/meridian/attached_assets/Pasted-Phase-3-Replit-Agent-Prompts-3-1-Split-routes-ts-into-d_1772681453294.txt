3.2 — Implement versioned Drizzle migrations
Meridian currently uses drizzle push for schema management, which provides no migration history and no rollback path. Switch to versioned migrations.
	1.	Install drizzle-kit if not already present.
	2.	Create or update drizzle.config.ts at the project root with the correct schema path and output directory (drizzle/migrations/).
	3.	Run drizzle-kit generate to create the initial migration snapshot from the current schema. This captures the existing 30-table schema as the baseline migration.
	4.	Commit the generated migration files to Git.
	5.	Update the app startup to run drizzle-kit migrate (or programmatic migration via drizzle-orm/migrator) before the server starts listening.
	6.	Remove or comment out any drizzle push calls in package.json scripts. Keep drizzle push available as db:push for emergency use only, but the default workflow is now db:generate → db:migrate.
Important: Do not alter any table definitions. This is a tooling change only — the schema stays identical. Take a mental note that a full DB backup should be taken before running this in production for the first time.

3.9 — Add password reset flow
Implement token-based password reset. This works in-app only for now (no email delivery yet — that’s 3.10).
	1.	Add a password_reset_tokens table: id, user_id, token (UUID), expires_at, used_at, created_at.
	2.	Add endpoint POST /api/auth/request-password-reset — accepts email, creates a token, returns the token in the response (since no email yet). In production with email, this would send the token via email and return only a success message.
	3.	Add endpoint POST /api/auth/reset-password — accepts token and new password, validates token not expired/used, updates password hash, marks token as used.
	4.	Add a “Forgot Password” link on the login page that shows a form and displays the reset token (temporary UX until email is wired).
	5.	Tokens expire after 1 hour. One active token per user — creating a new one invalidates the old one.
