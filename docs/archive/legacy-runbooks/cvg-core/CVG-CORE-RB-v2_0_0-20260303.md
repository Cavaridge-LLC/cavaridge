## 1. Entity Structure & IP Ownership

- **Cavaridge, LLC** is the sole owner of all intellectual property — applications, code, documentation, runbooks, and tooling.
- **Dedicated IT (DIT)** is a client and tenant. It is never a co-owner, co-developer, or embedded entity in any codebase.
- All clients, including DIT, are modeled as tenants within multitenant applications.
- No client name, logo, configuration value, or business logic may be hardcoded into any Cavaridge application.

## 2. Universal Build Standards

Every Cavaridge application must comply with these non-negotiable standards from day one:

| # | Standard | Requirement |
|---|---|---|
| 1 | **Multitenancy** | All apps are multitenant from the first line of code. No single-tenant assumptions. `tenant_id` on every data table. Row-level query scoping enforced. |
| 2 | **RBAC** | Role-based access control enforced at the data, API, and UI layer. Minimum taxonomy: Platform Owner, Platform Admin, Tenant Admin, User, Viewer. |
| 3 | **Theming** | Light, dark, and system modes required in all UI applications. System preference detection via `prefers-color-scheme`. Toggle persisted per user. |
| 4 | **No hardcoded client data** | All client config lives in tenant records, environment variables, or a database. No client names, rate cards, branding, or business logic in source code. |
| 5 | **IP hygiene** | All code and docs authored under Cavaridge, LLC ownership. LICENSE file in every repo. No DIT or client references in code. |
| 6 | **LLM access** | All in-app AI calls route through OpenRouter under the Cavaridge master key. No app-level LLM keys. Model selection in `llm.config.js` only. |
| 7 | **Secret management** | No plaintext keys in any repo. Replit Secrets for dev, Doppler for staging and production. `.env` gitignored before first commit. |
| 8 | **Rate limiting** | All API endpoints must have rate limiting applied. No unthrottled public routes. |
| 9 | **CSRF protection** | All state-changing endpoints protected against cross-site request forgery. |
| 10 | **Automated testing** | Baseline automated tests required for critical paths before any app is considered production-ready. |
| 11 | **CI/CD** | GitHub Actions pipeline for build validation. No deploy-by-button in production. |
| 12 | **Error handling** | Structured error handling on all API routes. React error boundaries on all client apps. Error monitoring (Sentry or equivalent) in staging and production. |

## 8. Runbook Extraction Prompts

### Claude Project Runbook Prompt

Run in each Claude project when generating or updating a runbook:

> "Generate a project state summary in markdown covering: (1) Project name and purpose, (2) Legal entity owner, (3) Tech stack and key dependencies, (4) Core features built or in progress, (5) Shared utilities or components that could apply to other apps, (6) Hardcoded values or assumptions that need to be made configurable, (7) Current RBAC and multitenancy status, (8) UI/UX standards in use (theming, component library, etc.), (9) Known gaps or technical debt, (10) Any Dedicated IT or client-specific references that need to be abstracted out."

### Replit Runbook Prompt

Run in each Replit project after significant build sessions:

> "Generate a technical state summary in markdown covering: (1) Project name and primary purpose, (2) Current tech stack — framework, language, runtime, key libraries and versions, (3) Folder and file structure — full tree with a one-line description of each key file or folder, (4) Environment variables currently in use — names only no values, (5) Authentication and authorization implementation — what exists what is missing, (6) Multitenancy status — implemented partial or absent, (7) UI theming — light/dark/system mode status, (8) Database schema or data model summary, (9) API endpoints or routes currently defined, (10) Third-party integrations — APIs SDKs services connected, (11) Hardcoded values that should be configurable — flag anything client-specific, (12) Known bugs incomplete features or technical debt, (13) What is production-ready vs. prototype/draft, (14) Deployment status — where hosted what is the deployment process."

### Synthesis Workflow (Bring Both to CVG-CORE)

1. **Merge & Reconcile** — combine both runbooks, flag contradictions
2. **Gap Analysis** — identify missing standards against Cavaridge build requirements
3. **Remediation Plan** — prioritized action list with effort estimates
4. **Portfolio Report** — roll up all apps into a single health status view

## 10. OpenRouter — Centralized LLM Gateway

**Rule:** No app holds its own LLM API key. All in-app AI calls route through OpenRouter under the Cavaridge master key.

### Why OpenRouter

- Single billing account under Cavaridge, LLC
- Swap models without touching app code
- Per-app and per-tenant usage tracking
- Access to Claude, GPT-4, Gemini, Mistral, and others from one endpoint

### Model Routing — Which LLM for What

| Task Type | Model | Rationale |
|---|---|---|
| Structured report generation | anthropic/claude-sonnet-4-6 | Deterministic, instruction-following |
| Risk scoring / classification | anthropic/claude-sonnet-4-6 | Consistent output format |
| Long document analysis | google/gemini-pro-1.5 | Large context window |
| Fast lightweight / UI hints | mistralai/mistral-7b-instruct | Low latency, low cost |
| Code generation (in-app) | anthropic/claude-sonnet-4-6 | Best code quality |
| Embeddings / semantic search | openai/text-embedding-3-small | Cost-effective at scale |
| Vision / image analysis | openai/gpt-4o | Multi-modal capability |

### Central Config Pattern (All Apps)

```javascript
// llm.config.js — Cavaridge standard
export const LLM_ROUTES = {
  reportGeneration:   "anthropic/claude-sonnet-4-6",
  riskClassification: "anthropic/claude-sonnet-4-6",
  documentAnalysis:   "google/gemini-pro-1.5",
  lightweightUI:      "mistralai/mistral-7b-instruct",
  codeGeneration:     "anthropic/claude-sonnet-4-6",
  embeddings:         "openai/text-embedding-3-small",
  vision:             "openai/gpt-4o",
}
```

Model selection lives only in this file — never inline in a component or API route.

### Tenant Cost Tracking

Tag every OpenRouter request with `app_code` and `tenant_id` in request metadata to enable per-tenant AI cost reporting and future billing.

### Current Non-Compliance

| App | Issue |
|---|---|
| CVG-CAELUM | Uses `OPENAI_API_KEY` env var pointing to OpenRouter — misleading name |
| CVG-ASTRA | Model ID hardcoded in routes.ts (`anthropic/claude-sonnet-4`, temp 0.4, max 8192) |
| CVG-MIDAS | OpenRouter planned but not implemented |
| CVG-VESPAR | No AI integration yet |
| CVG-CERES | Uses OpenRouter via OpenAI SDK — correct approach, needs `llm.config.js` |

## 11. Secret Management

**Rule:** No plaintext keys in any repo, ever.

### By Environment

| Environment | Tool |
|---|---|
| Replit dev | Replit Secrets (padlock icon) — never paste keys in code or chat |
| Staging & Production | Doppler — syncs to Replit, Vercel, and GitHub Actions from one dashboard |
| Enterprise / backup | HashiCorp Vault or AWS Secrets Manager |

### Doppler Setup

1. Create Doppler account under Cavaridge, LLC
2. Create a project per app (cvg-hipaa, cvg-meridian, etc.)
3. Add environments: development, staging, production
4. Sync to Replit, Vercel, and GitHub Actions via native integrations
5. Store master OpenRouter key in Doppler only — never share, never paste

### Hard Rules

- `.env` gitignored on every repo before the first commit — verified before any push
- No keys in Claude conversations, Replit Agent chat, GitHub issues, or any communication tool
- Master OpenRouter key stays in Doppler only — apps get scoped sub-keys
- Sub-keys have minimum required scope — inference only, no billing access
- Rotate immediately on any suspected exposure

### Standard .env Template (Dev Reference Only)

```
TENANT_ID=
APP_ENV=development|staging|production
AUTH_PROVIDER=
DB_URL=
THEME_DEFAULT=system
OPENROUTER_API_KEY=
LLM_CONFIG_PATH=./llm.config.js
```

## 17. RBAC Role Taxonomy (Portfolio Standard)

Minimum roles every app must implement. Apps may extend with additional roles but may not omit these.

| Role | Scope | Capabilities |
|---|---|---|
| **Platform Owner** | Global | Full control over all tenants, settings, billing, platform config |
| **Platform Admin** | Global | Manage tenants, impersonate tenant admins, no billing or platform settings |
| **Tenant Admin** | Tenant | Full control within their tenant — users, settings, data, config |
| **User** | Tenant | Create, read, update within their tenant. No admin functions. |
| **Viewer** | Tenant | Read-only access within their tenant |

Meridian's 7-tier model with 21 permissions and deal-level access is the reference implementation. Other apps should align progressively toward this maturity level.
