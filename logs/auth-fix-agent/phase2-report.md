# Phase 2 — CVG-AI (Spaniel) Auth Wiring Report

**Date:** 2026-03-24
**App:** CVG-AI / Spaniel (`apps/spaniel/`)
**Build order position:** 1 (first app)

---

## Summary

Spaniel is a service-to-service LLM gateway — no end-user sessions. Auth wiring focused on integrating `@cavaridge/auth` for shared utilities while preserving the service-token authentication model appropriate for an internal gateway.

---

## Changes Made

### 1. Added `@cavaridge/auth` dependency

- **`package.json`**: Added `"@cavaridge/auth": "workspace:*"` to dependencies.
- **`tsconfig.json`**: Added path mappings for `@cavaridge/auth` and `@cavaridge/auth/*`.

### 2. Rewrote `server/middleware/auth.ts`

**Before:** Fully local implementation — manual `Authorization` header parsing, regex-based Bearer extraction, token validation against `SPANIEL_SERVICE_TOKENS`.

**After:** Imports `extractBearerToken` from `@cavaridge/auth/server` for standardized token parsing. Defines `ServiceRequest` interface extending `express.Request` with `serviceId` field for service caller identification. Token validation logic (checking against `SPANIEL_SERVICE_TOKENS`) remains — this is correct for a service-to-service gateway that doesn't use Supabase user sessions.

**Rationale for keeping service tokens:** Spaniel is called by other Cavaridge services (primarily Ducky), not by end users. `createAuthMiddleware` from `@cavaridge/auth` is designed for cookie/bearer user sessions, which is not the right auth model here. The shared `extractBearerToken` utility is reused; the token validation is Spaniel-specific.

### 3. Updated all route handlers to use `ServiceRequest`

All route files now import `ServiceRequest` from `../middleware/auth.js` instead of using plain `express.Request`:
- `server/routes/reason.ts` — POST `/api/v1/reason`, POST `/api/v1/embed`
- `server/routes/usage.ts` — GET `/api/v1/usage`
- `server/routes/models.ts` — GET `/api/v1/models`, POST `/api/v1/models/refresh`
- `server/routes/health.ts` — GET `/api/v1/health`, GET `/healthz`, POST `/api/v1/migrate`

### 4. Enforced tenant scoping on all queries

- **`/api/v1/reason`**: Already enforced — Zod schema requires `tenant_id` (string, min length 1). No change needed.
- **`/api/v1/embed`**: Already enforced — Zod schema requires `tenant_id`. No change needed.
- **`/api/v1/usage`**: **Previously optional** — `tenant_id` was an optional query parameter, allowing unscoped queries across all tenants. **Now required** — returns 400 if `tenant_id` is not provided. The `eq(requestLog.tenantId, tenantId)` condition is always applied.
- **`/api/v1/models`**: No tenant-specific data (routing matrix is platform-global). No scoping needed.
- **`/api/v1/health`**: No tenant-specific data. No scoping needed.
- **`/api/v1/migrate`**: Schema migration — platform-level operation. No scoping needed.

### 5. No local auth logic removed (was already minimal)

Spaniel had no duplicate `loadUser()` or `createAuthMiddleware` copies. The only local auth was the service token middleware, which was refactored to use `@cavaridge/auth` utilities.

# Phase 2 — CVG-CAELUM (Caelum) Auth Wiring Report

**Date:** 2026-03-24
**App:** CVG-CAELUM / Caelum (`apps/caelum/`)
**Build order position:** 3 (after Ducky)

---

## Summary

Caelum is the SoW builder — an MSP-level app requiring MSP Admin + MSP Tech minimum access. Auth was partially wired via `@cavaridge/auth` but had significant local duplications: a local `loadUser` middleware, a full local RBAC system with its own `roles`/`userRoles`/`userTenants` tables, a local `tenantScope` middleware with hardcoded DIT default tenant slug, and non-standard role names (`Platform Owner`, `Tenant Admin`, `User`, `Viewer`). All local auth logic removed; everything now flows through `@cavaridge/auth`.

---

## Changes Made

### 1. Replaced local `loadUser` with shared `createAuthMiddleware`

**Before:** `server/services/auth/middleware.ts` had a local `loadUser()` function that duplicated the shared middleware — manually called Supabase, loaded profile, resolved tenant, checked `isActive`.

**After:** `server/index.ts` now imports `createAuthMiddleware` from `@cavaridge/auth/server` and applies it globally with `db`, `profiles`, and `tenants` tables. The local middleware file is reduced to a thin re-export of `requireAuth` and `AuthenticatedRequest`.

### 2. Fixed auth routes config

**Before:** `registerSharedAuthRoutes` was called with `orgsTable: tenants` (wrong property name) and `defaultRole: "user"` (non-standard role).

**After:** Corrected to `tenantsTable: tenants` and `defaultRole: "msp_tech"` (UTM standard).

### 3. Removed local RBAC middleware

**Before:** `server/middleware/rbac.ts` had a full local implementation — `ROLE_NAMES` constant with 10 roles including legacy names (`Platform Owner`, `Tenant Admin`, `User`), numeric hierarchy, `loadUserRole()` function querying local `roles`/`userRoles` tables with in-memory cache, and a local `requireRole()` factory.

**After:** File reduced to thin re-exports of `requireRole` and `ROLES` from `@cavaridge/auth/guards` and `@cavaridge/auth`. The shared `requireRole` uses `hasMinimumRole()` against the standard 6-role UTM hierarchy.

### 4. Removed local `tenantScope` middleware

**Before:** `server/middleware/tenantScope.ts` had local tenant resolution logic — queried `userTenants` table, fell back to hardcoded `DEFAULT_TENANT_SLUG = "dedicated-it"` (DIT-specific violation of CLAUDE.md), resolved `tenantType` and `mspTenantId` via recursive CTE. Also augmented global `Express.Request` with `tenantId`, `tenantType`, `mspTenantId`.

**After:** File reduced to a no-op passthrough function. Tenant resolution is now handled by the shared `createAuthMiddleware` which sets `req.tenant`, `req.tenantId`, and `req.accessibleTenantIds`. The hardcoded DIT slug is removed.

### 5. Updated all route guards to UTM roles

Replaced all route middleware chains from `requireAuth, tenantScope, loadUserRole, requireRole(ROLE_NAMES.XXX)` to simplified `requireAuth, requireRole(ROLES.XXX)`.

**Role mapping applied:**

| Old Role | New UTM Role | Route Type |
|----------|-------------|------------|
| `ROLE_NAMES.VIEWER` | `ROLES.MSP_TECH` | Read endpoints (conversations, versions, models, exports) |
| `ROLE_NAMES.USER` | `ROLES.MSP_TECH` | Write endpoints (chat, SoW edit, branch, flag, title, grammar check, Ducky push) |
| `ROLE_NAMES.TENANT_ADMIN` | `ROLES.MSP_ADMIN` | Delete endpoints (conversations, messages) |

### 6. Removed local auth tables from shared schema

**Before:** `shared/models/auth.ts` defined local `userTenants`, `roles`, and `userRoles` Drizzle tables alongside re-exports from `@cavaridge/auth/schema`.

**After:** Local tables removed. File now re-exports only `profiles`, `tenants`, `tenantMemberships` and type aliases from `@cavaridge/auth/schema`.

### 7. Updated seed.ts

**Before:** Referenced local `roles`, `userRoles`, `userTenants` tables. Hardcoded "Dedicated IT" as default tenant slug.

**After:** Uses `profiles`, `tenants`, `tenantMemberships` from `@cavaridge/auth/schema`. Creates a "Cavaridge" platform tenant and a "Demo MSP" tenant. Maps users via `tenantMemberships` with `msp_admin` role.

### 8. Updated client-side auth

**Before:** `client/src/hooks/use-auth.tsx` had non-standard role permissions map using `platform_owner`, `tenant_admin`, `user`, `viewer`.

**After:** Updated to UTM 6-role permissions map: `platform_admin`, `msp_admin`, `msp_tech`, `client_admin`, `client_viewer`, `prospect`.

### 9. Updated home.tsx role checks

**Before:** Role comparisons used display names (`"Viewer"`, `"Platform Owner"`, `"Tenant Admin"`).

**After:** Uses UTM enum values (`"client_viewer"`, `"prospect"`, `"platform_admin"`, `"msp_admin"`). Also fixed `profileImageUrl` → `avatarUrl` to match shared `Profile` type.

### 10. Fixed pre-existing type errors

- `sowExport.ts` — Fixed `children: Paragraph[]` → `(Paragraph | Table)[]` for docx table generation
- `routes.ts` — Fixed Express 5 `req.params` type (`string | string[]` → cast to `string`)
- `routes.ts` — Fixed `MODEL_ROSTER` const array union type narrowing in `pickBestModel`/`pickEnsembleModels`
- `fileExtractor.ts` — Fixed `pdf-parse` dynamic import with `as any` cast
- `db.ts` — Fixed `pg.PoolClient` namespace reference → `any`
- `tenantConfigLoader.ts` — Fixed `configJson` → `config` (shared Tenant type field name)
- `storage.ts` — Fixed dead import of `authStorage` → re-export `chatStorage`
- Added `server/declarations.d.ts` for `pg` and `pdfkit` module declarations
- Added `server/types.ts` for global Express.Request augmentation (auth properties)
